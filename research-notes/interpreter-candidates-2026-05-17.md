# Interpreter candidates — Babel 2026-05-17

Generated by research agent. Babel currently runs `brainfuck_tape` only; the schema validates broader. This file surveys (Task 1) BF-family candidates Babel could cover today via parameter sheets, and (Task 2) non-tape families Babel could cover after runtime extensions, with feasibility estimates.

The schema as of this writing accepts these `BaseMachine` values — `brainfuck_tape`, `stack`, `oisc`, `fungeoid_2d`, `register`, `queue`, `string_rewriting` — but only the first has an interpreter behind it (`src/babel/interpreter.py`). For Task 1, "Babel-fit = high" means the existing schema + interpreter can run a parameter sheet today; "Babel-fit = mid" means a parameter sheet works but a small schema extension would be cleaner (e.g. a new `InstructionOp` enum value); "Babel-fit = low" means a non-trivial schema change is required even though the language is still tape-based; "impossible" means it isn't tape-shaped at all.

## Task 2: Non-tape families — runtime extension feasibility

For each non-tape `BaseMachine`, an effort estimate, the canonical languages it would unlock, and what's hard.

### Stack-machine family (`BaseMachine.STACK`)

**Canonical languages:**
- **Forth** — the canonical stack language; whitespace-separated word tokens; dictionary semantics.
- **FALSE** (https://esolangs.org/wiki/FALSE) — Wouter van Oortmerssen, 1993; stack + 26 named variables (a-z); lambdas; influenced Brainfuck and Befunge. Single-character ascii encoding.
- **Underload** (https://esolangs.org/wiki/Underload) — 8 commands; *stack of strings* (each stack element is itself a program text), with `^` to execute. ais523, 2006. Stack model but the element type is text rather than int.
- **Whitespace** (https://esolangs.org/wiki/Whitespace) — stack + heap; whitespace-only encoding; arithmetic, flow control.
- **Joy / PostScript** — concatenative; outside the esolang corpus but standard exemplars.

**Runtime-engineering effort: small (a long weekend).** A stack machine is the smallest delta from a tape machine: replace the tape + pointer with a list/deque, change the dispatch table from `INCREMENT/DECREMENT/PTR_*` to `PUSH/POP/DUP/SWAP/ROT/ADD/SUB/MUL/DIV/EQ/IF/WHILE`. The control-flow model (loops by tokens, conditional execution) is similar in shape to BF's `[`/`]`. Babel's `MemoryShape.STACK_UNBOUNDED`/`STACK_BOUNDED` already validate.

**Architectural blockers:**
- `InstructionOp` is currently tape-specific. Need a parallel enum (or a discriminated union) for stack ops: `PUSH_LITERAL`, `POP`, `DUP`, `SWAP`, `ROT`, `DROP`, `OVER`, `PICK`, `ADD`, `SUB`, `MUL`, `DIV`, `MOD`, `EQ`, `LT`, `GT`, `NOT`, `IF`, `WHILE`, `CALL`, `RETURN`, plus I/O. The schema's `_TAPE_OPS`/`_CANONICAL_BF_OPS` split is a precedent for this — add `_STACK_OPS`/`_CANONICAL_STACK_OPS`.
- Literal pushing requires the language to distinguish "this token is an instruction" from "this token is a literal value". Either the instruction set declares a numeric-literal pattern (`r"-?\d+"` for FALSE, digit characters for Befunge), or every literal value gets its own instruction (workable for tiny languages like Underload where all literals are strings in parens).
- Named variables (FALSE's a-z) need `MemoryShape.NAMED_VARIABLES` to be made legal for `BaseMachine.STACK` (currently only `STACK_UNBOUNDED`/`STACK_BOUNDED`).
- Underload requires *stack-of-strings* semantics; that's a different cell type (string) from FALSE's stack-of-ints. Either parameterize cell type per language or add a `STRING` cell width.

**Strategic value:** Highest among non-tape families. Stack languages are the second-largest esolang population after BF derivatives, and the canon (Forth, FALSE, Underload, Whitespace) is mature and well-specified. Shipping the stack family makes Babel a credible general parameter-driven runtime instead of "a BF parameter system".

### Fungeoid 2D family (`BaseMachine.FUNGEOID_2D`)

**Canonical languages:**
- **Befunge** / **Befunge-93** / **Funge-98** (https://esolangs.org/wiki/Befunge) — the canonical 2D stack machine; 80×25 grid (93), unbounded (98); self-modification via `g`/`p`.
- **Fish (`><>`)** (https://esolangs.org/wiki/Fish) — Befunge-inspired; mirrors, nested stacks, wrapping grid.
- **Piet** (https://esolangs.org/wiki/Piet) — 2D color-block source; instructions = color transitions; needs an image loader.
- **Wierd** — 2D, direction by character adjacency (not a literal symbol grid).
- **Cardinal**, **Cubix**, **Befunk**, **Flobnar** — 2D variants.

**Runtime-engineering effort: medium (a couple weeks).** The instruction set is small (Befunge-93 is ~30 instructions) but the execution model is fundamentally different: instruction pointer is `(x, y, direction)` instead of an integer. String mode adds a second dispatch mode. Self-modification (`p`) means the source grid is a mutable runtime state, not a static input — this changes how `loader.py` should hand source to the interpreter.

**Architectural blockers:**
- `MemoryShape.GRID_2D_TORUS` exists but the interpreter has no concept of an `(x, y)` cursor with `(dx, dy)` direction. New runtime primitive.
- `Encoding.TWO_DIMENSIONAL_GRID` exists and is validated, but the loader treats source as text-and-tokenize; needs a grid loader that preserves rows.
- Two coexisting memories per language (the 2D source grid + the stack). Babel's current `LanguageSpec.memory_shape` is a single field. A 2D fungeoid needs *both* `GRID_2D_TORUS` (the source/playfield) *and* `STACK_UNBOUNDED` (the runtime). The schema's `_LEGAL_MEMORY_FOR_BASE` already pairs `FUNGEOID_2D` with both shapes, but the field is singular — either the runtime infers "if base is fungeoid then both are present" or the schema gets a list-typed `memory_shapes` for these cases.
- Piet specifically needs a non-text source (PNG/GIF/PPM) and a codel-detection pre-pass; that's its own image loader.

**Strategic value:** Befunge-family is the second-largest 2D-language cluster on the wiki (532 entries in `Category:Two-dimensional_languages` per the survey). Covering Befunge-93 + Fish would let Babel claim 2D coverage. Piet is iconic and gets disproportionate media attention (visual programming), so adding it is a strong publication hook even though it's only one language.

### OISC family (`BaseMachine.OISC`)

**Canonical languages:**
- **Subleq** (https://esolangs.org/wiki/Subleq) — subtract-and-branch-if-≤0; three-address; memory-mapped I/O at address -1. Most-cited OISC.
- **BitBitJump** (https://esolangs.org/wiki/BitBitJump) — copy-bit-and-jump; bounded operand → bounded-storage machine.
- **ByteByteJump** — byte-version of BitBitJump.
- **FlipJump** — flip-bit-and-jump.
- **RSSB** — reverse-subtract-and-skip-if-borrow; instruction pointer is memory-mapped.
- **SBN** (and SBN2/3/4) — subtract-and-branch-if-negative; Subleq-adjacent.

**Runtime-engineering effort: small (a long weekend).** The instruction-set count is one. The complexity is all in addressing modes (absolute vs. relative; memory-mapped IP vs. distinct; I/O via reserved addresses). Babel's `MemoryShape.NAMED_VARIABLES` already validates for `OISC`. A Subleq interpreter is ~50 lines.

**Architectural blockers:**
- `InstructionOp` is tape-specific; need a single `OISC_INSTRUCTION` op whose semantics are parameterized by an "instruction spec" sub-model (operand count, what each operand does, condition). This is a generalization the current enum doesn't model.
- Source parsing: OISC programs are usually whitespace-separated integers or labeled-assembly. Babel's `Encoding` enum doesn't have an "integer literals with optional labels" mode; closest is `whitespace_separated_tokens`.
- I/O via reserved addresses (Subleq's address -1) is a runtime convention that needs to be declarable per parameter sheet.
- Memory model is *uniform memory* (program and data share the same address space). The interpreter would need to load the program into memory rather than treat it as a separate token stream.

**Strategic value:** Small population (117 entries in `Category:OISC`) but very-often-cited as a *minimality demonstration*. Babel covering Subleq would let the methodology paper claim, with one parameter sheet, "the smallest non-trivial language we describe has one instruction". High narrative value per line of runtime code.

### Register-machine family (`BaseMachine.REGISTER`)

**Canonical languages:**
- **Minsky machine / counter machine** (https://esolangs.org/wiki/Counter_machine) — 2+ unbounded registers, increment/decrement-if-nonzero-else-branch. The canonical proof-of-Turing-completeness register model.
- **FRACTRAN** — single register holding a positive integer; program is a list of fractions; semantics is "multiply by the first fraction that yields an integer". Conway, 1972.
- **Wang B-machine** — historical register-machine variant.
- **TC** (`Three Counter`) — Minsky-equivalent.
- **Random brainfuck-derived register-style languages** — rare; most "register" entries on the wiki are toy x86-likes.

**Runtime-engineering effort: small (a long weekend) for Minsky-style, medium for FRACTRAN.** Minsky is trivial: a dict of named registers, two ops (`INC reg` and `DEC_OR_BRANCH reg label`), labels. FRACTRAN is conceptually trivial but requires arbitrary-precision arithmetic (Babel has `CellWidth.ARBITRARY` — good) and a rational-number type.

**Architectural blockers:**
- `InstructionOp` needs `REG_INCREMENT`, `REG_DECREMENT`, `BRANCH_IF_ZERO`, plus a `register` field on `Instruction` (currently `Instruction` has only `token`/`op`/`description` — no operand slot).
- Branching by label means programs have an instruction-address space, not just a token stream. The loader would need a label-resolution pass.
- FRACTRAN is structurally different from Minsky — instructions are *fractions*, not named ops. A FRACTRAN sheet wouldn't fit a Minsky template; this is a small family but heterogeneous.

**Strategic value:** Moderate. Minsky/Counter machines are a CS-classics nod but esolang.org's `register` category is thin. Strongest argument: pedagogical — Babel could claim coverage of "the three classic Turing-complete models" (Turing-tape ≈ BF, stack ≈ Forth, counter ≈ Minsky).

### Queue family (`BaseMachine.QUEUE`)

**Canonical languages:**
- **Queue** (https://esolangs.org/wiki/Queue) — wiki landing page for the model.
- **Qdeql** (https://esolangs.org/wiki/Qdeql) — single queue of bytes; explicitly modeled as "the queue equivalent of BF".
- **Fueue** — single queue, ais523.
- **Queueue** (https://esolangs.org/wiki/Queueue) — nested queues.
- **Q-BAL** — "stack-based, but with queues".
- **Eniuq** (https://esolangs.org/wiki/Eniuq) — stack + queue hybrid.
- **NULL** — three queues.
- **Huit** (https://esolangs.org/wiki/Huit) — single queue of floats or strings.

**Runtime-engineering effort: small (a long weekend).** A queue interpreter is structurally a stack interpreter with `pop` replaced by `popleft`. Babel already has `MemoryShape.QUEUE`/`DEQUE` validating for this base.

**Architectural blockers:**
- Same `InstructionOp` extension problem as stack — need `ENQUEUE`, `DEQUEUE`, possibly `PEEK_FRONT`, `PEEK_BACK` for deques. Likely shares 60-70% of ops with the stack family.
- Cell type variation: Huit allows floats *or* strings per element. Babel's `CellWidth` enum is integer-shaped; floats/strings need a `cell_type` (separate from cell width).

**Strategic value:** Lowest among non-tape families. Queue languages are rare (~10 wiki entries vs. 1,000+ stack-based, 800+ BF derivatives). Covering them is cheap *if* the stack runtime already exists (shared op set), so the right move is "ship queue support as a free byproduct of the stack family, not as a standalone investment."

---

### String-rewriting family (`BaseMachine.STRING_REWRITING`)

**Canonical languages:**
- **Thue** (https://esolangs.org/wiki/Thue) — nondeterministic; rule list + initial state; `::=` syntax; I/O via reserved RHS markers (`:::`, `~`).
- **///** (slashalash, https://esolangs.org/wiki////) — deterministic, single-rule substitution; print-by-default; no input.
- **Markov algorithm** (https://esolangs.org/wiki/Markov_algorithm) — deterministic; rule order matters; first-match-first-rule rewriting.
- **Tag systems** / **cyclic tag systems** — historic variants.
- **Bitwise Cyclic Tag** — minimal string-rewriter; the proof carrier for ///'s Turing-completeness.

**Runtime-engineering effort: medium (a couple weeks).** Mechanics are simple but the *programming model is alien to Babel's instruction-list assumption*. A Thue program isn't a sequence of token→op mappings; it's a set of rewrite rules `lhs → rhs`. Babel's `Instruction` model (token + canonical op) doesn't fit. Need a parallel `RewriteRule` model and a new entry in `LanguageSpec` (or a discriminated union on `instructions`).

**Architectural blockers:**
- There is no instruction pointer in the BF/Befunge sense — every step is a rule-search over the entire current state string. This is a runtime-shape change.
- `InstructionOp` is irrelevant for this family — the canonical operation *is* "find LHS, replace with RHS". The dispatch table abstraction doesn't apply.
- Nondeterminism (Thue) needs a rule-selection policy (random? lexical? first-match?). That's a per-language declaration not currently in the schema.
- Halting conditions are "no rule matches" — needs a runtime convention.

**Strategic value:** Methodological. Thue and /// are *canonical* in the esolang literature for "computation without an instruction pointer". Covering them lets Babel claim it spans *both* control-flow paradigms (instruction-stream-driven AND state-search-driven). The schema work is non-trivial, but only the *schema* — the runtime is small.

---

### Recommendation — best first Path B investment

**Ship the stack family first.** Reasons:

1. **Cheapest runtime delta.** A stack interpreter is the smallest structural change from the tape interpreter (replace tape+pointer with deque; replace `INCREMENT`/`DECREMENT`/`PTR_*` with `PUSH`/`POP`/`DUP`/`SWAP`).
2. **Highest population to leverage.** `Category:Stack-based` has 1,003 wiki entries; stack-machine canon includes Forth, FALSE, Underload, Whitespace — all of which are well-specified and have stable references.
3. **Queue family is a near-free byproduct.** Once the stack runtime exists, swapping `pop` for `popleft` covers the queue family for ~10% extra effort.
4. **Unlocks Befunge later.** Befunge is fungeoid-2D *plus* a stack; building the stack machinery first makes the eventual fungeoid extension easier (the fungeoid runtime is "2D-IP + stack ops", and the stack ops will already exist).
5. **Strongest methodological claim.** With BF-tape + stack covered, Babel can credibly say "we describe and execute two of the three classical Turing-complete models from a single parameter format". That's the publication-worthy moment.

Secondary recommendation: **Subleq (OISC family) as the runner-up.** Cheapest in absolute terms (~50 lines of interpreter), highest narrative-density per line (the "one instruction" headline), and small enough that it could be done as a follow-up appendix to a stack-machine installment without earning its own white paper.

Defer: **fungeoid-2D** (medium effort, depends on stack), **string-rewriting** (alien programming model, biggest schema rework), **register** (small but thin canon).

## Methodology notes

**How the survey was conducted.** WebFetch against `https://esolangs.org/wiki/` pages, prioritizing the language pages named in the task brief (Ook!, Spoon, Mierda, La Weá, Chespirito) plus a breadth-first sweep of `Category:Brainfuck_derivatives`, `Category:Trivial_brainfuck_substitution`, `Category:Unimplemented` (BF-relevant entries), and the canonical Path B categories (`Category:Stack-based`, `Category:Two-dimensional_languages`, `Category:OISC`). The esolangs.org categories surface ~200 entries per page; the BF derivatives category alone has 806 entries, so this survey is illustrative, not exhaustive.

**What was skipped and why.**

- The full 806-entry BF derivative list. Beyond the canonical and culturally-significant ones surveyed (Task 1A/1B/1C), the long tail is dominated by single-author joke languages with vanishing cultural reach (`PenisScript`, `UwU`, `(1)_Grace_sent_you_a_message`, `4Head`, etc.). Babel covering them would be busywork rather than methodologically meaningful.
- 2D BF variants (Brainloller, Paintfuck, ABrainloller, BrainSpace 1.0). These need either image-source loaders or hybrid tape+2D-source semantics that the current schema can't express; they were flagged with "skip" / "impossible" but cataloged so Path B planning can revisit them.
- Multithreading / time-travel BF variants (Brainfork, 5D BF). Fundamentally runtime-feature additions, not parameter-sheet candidates.
- Detailed token-by-token mapping for every recommended sheet — the parameter sheets themselves are the deliverable; this survey only confirms feasibility.

**Verifiability caveats.**

- Interpreter status was assessed by reading the wiki page and noting linked implementations; I did not click through to GitHub to confirm repo activity dates, except where the wiki itself summarized status (e.g. Roadrunner "last updated June 2023"). "Dead link" / "obscure" claims are based on the wiki text, not direct verification.
- The wiki's own implementation status fields can lag reality in both directions (some languages marked "unimplemented" have community impls posted in comments; some "implemented" claims point to defunct repos). For any candidate before committing a parameter sheet, do a 30-second link check.
- Cell-width and I/O assumptions for under-specified languages (e.g. Pikalang says "Not specified") were treated as "inherits BF defaults" — that's the wiki's convention, but a parameter-sheet author should still pick explicitly.

**Coverage confidence by sub-population.**

- 1A (unimplemented BF-family): Light. There are ~1,400 unimplemented esolangs and the wiki only surfaces 200 per category page. I sampled 8 from the visible page. A deeper trawl (paginating through `Category:Unimplemented` ∩ `Category:Brainfuck_derivatives` cross-reference) would likely 5×-10× this list.
- 1B (abandoned/obscure interpreters): Moderate. The wiki tends to mark interpreter status reasonably; I covered the prominent cases (Common Lisp / Ruby one-offs / fragile platform builds). Long tail unknown.
- 1C (culturally significant): High. The canonical list is small (Ook!, Spoon, Mierda, La Weá, Chespirito, COW, Pbrain, Boolfuck, Smallfuck, GERMAN, Pikalang, BaguaFuck, plus a few more) and is covered.
- Path B canonical languages: High for stack/fungeoid/OISC/string-rewriting (the canon is small and well-known); moderate for register (the wiki's `register` category is thin and conflates "named-variable VM" with "x86-like esolang"); low for queue (only 8-10 entries total, but small enough that the survey is plausibly complete).
