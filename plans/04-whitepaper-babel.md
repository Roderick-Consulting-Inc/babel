## 1. A field that has produced examples but not a method

An esoteric programming language is a programming language designed to test the boundaries of what a programming language can be: usually small, often deliberately impractical, sometimes a joke that turns out to be a research programme on second reading. The community at esolangs.org [@esolangs_wiki] has been cataloguing them for over two decades. The wiki currently lists something on the order of 1,500 entries. Its single largest category is *Brainfuck derivatives* [@esolangs_bf_derivatives], with around eight hundred languages — each one a re-encoding of the same eight-instruction tape machine that Urban Müller distributed in 1993 [@mueller_brainfuck_1993] under a different surface form, a different cell width, a different alphabet, or a different cultural skin.

Read sympathetically, this is a remarkable creative output. Read carefully, it is also a strange artefact. Eight hundred authors took the same starting point and did the same kind of restructuring work, mostly without referring to each other's tools or inheriting each other's scaffolding. The field has produced a corpus but not a methodology. It has discovered the variation axes its languages move along — cell width, tape topology, instruction count, surface vocabulary, cultural theme — without ever consolidating them into a parameter schema that a new author could pick up and use.

The closest analogues elsewhere in computing are the long-running obfuscation contests — the International Obfuscated C Code Contest, running annually since 1984 [@ioccc], and the Obfuscated Perl Contest of the late 1990s [@obfuscated_perl_contest]. Those communities accept programs that are formally correct, executable, and unreadable without dedicated study. They explore the same gap that esoteric languages do — the gap between *what compiles* and *what communicates* — but from inside an existing language. The esolang field plays the same game from outside: it builds new languages that hold the gap open by design. Both communities have learned, in their own ways, what kinds of constraint produce interesting strangeness. Neither has produced a tool that would let the next contributor inherit that learning.

This paper proposes that tool. Babel is a methodology for the programmatic construction of esoteric programming languages, plus the parameter schema and runtime architecture that make the methodology operable. The argument is that making the field's tacit variation axes explicit — and providing a runtime that turns chosen parameter values into a working language — changes the economics of esoteric-language design. It is a lever on the *next* eight hundred attempts, not on the existing eight hundred.

## 2. What an esoteric language is, for the purposes of this paper

The wiki at esolangs.org takes a generous view of what counts as esoteric. The category includes Turing-complete languages, sub-Turing languages, deliberately undecidable jokes, languages that exist as a single program, and languages that are formally identical to existing ones with one or two cosmetic substitutions. For the purposes of this paper, an esoteric programming language is one with at least the following features.

It has an executable semantics — there is a clear answer, in principle, to the question *what does this program do?* This excludes pure performance art and language-as-poetry projects, however adjacent they may be culturally. It has a small surface area — typically a few dozen instructions or fewer, often fewer than ten. This excludes the heavyweight production languages (Java, Python) and the small but architecturally rich ones (Forth, Lua) that aim to be genuinely useful at scale. And it has a deliberate constraint or strangeness — the language was designed to be in some way *interesting* rather than *useful*, where the interest comes from a chosen restriction, addition, or framing that production languages would not adopt.

These criteria admit Brainfuck (eight instructions, deliberate minimalism), Whitespace (only spaces and tabs are syntactic), Befunge (two-dimensional source grid), Shakespeare (English-language plays as syntax), Wenyan (Classical Chinese grammar as syntax) [@huang_wenyan], INTERCAL (deliberately user-hostile error messages), and Piet (program-as-bitmap). They exclude general-purpose languages with eccentric communities (Lisp, Smalltalk), code-golf languages whose goal is shortness rather than constraint (J, K), and pure jokes with no executable semantics.

The criteria are not particularly novel. The community has converged on something like them implicitly. Stating them explicitly matters because the parameter schema in the next section depends on having a definite class of artifact to parameterise. A schema that tries to cover everything — production languages, code-golf languages, executable poetry, and esolangs proper — would be too loose to be useful. Babel parameterises the class of language that meets the three criteria above.

## 3. The variation axes the field has implicitly used

The eight hundred Brainfuck derivatives, taken as a sample, vary along a small number of axes. Reading widely across the corpus suggests the following partition.

**The base machine.** Every esoteric language has an underlying execution model — the abstract machine its programs run on. The corpus uses, in approximate order of frequency: tape machines (Brainfuck and its derivatives), stack machines (FALSE, GolfScript), one-instruction-set machines (Subleq, BitBitJump), two-dimensional grid machines (Befunge, Hexagony), register-and-jump assembly clones, queue machines (Fueue), string-rewriting systems (Thue, Markov-style), and a handful of more exotic machines (combinator calculus, cellular automata as program). The base machine is the largest single design choice; everything else parameterises it.

**Memory shape.** Within a given base machine, the memory has a shape: a one-dimensional unbounded tape, a one-dimensional circular tape, a two-dimensional torus, a stack of fixed depth, an unbounded stack, a queue, a deque, a graph. Brainfuck's standard memory is a one-dimensional unbounded tape of byte-sized cells; Boolfuck uses a tape of bit-sized cells; Brainfuck Extended uses arbitrary-precision integers.

**Cell width.** Closely related but separable: the size of an individual cell. Bit, nibble, byte, word, arbitrary precision. Some languages allow signed values, some unsigned, some only positive.

**Instruction count and set.** A Brainfuck derivative might use the canonical eight instructions, a subset (Boolfuck), a superset (Brainfuck with debug or randomness), or a non-overlapping replacement. The instruction count typically ranges from one (OISC machines) to a few dozen (richer fungeoids). Larger instruction sets are usually re-encodings of small ones rather than truly novel additions.

**Encoding.** Once an instruction set exists, it has to be encoded in source text. The canonical Brainfuck encoding uses ASCII punctuation: `> < + - . , [ ]`. Whitespace encodes its instructions in tab/space/newline triples. Unary encodes its entire program as a single integer expressed in base-1. Binary, base-N, and natural-language-keyword encodings are all options.

**I/O model.** Character at a time is the dominant pattern. Some languages take whole-line input, some take an integer, some take no input and produce no output (the program's effect is its own internal state at termination), some treat the file system as I/O.

**Additions and removals.** Many derivatives are characterised primarily by what they add to or subtract from a base. La Weá adds a clipboard register to Brainfuck's eight instructions. Chespirito adds a randomness instruction. Brainfuck Lite removes the loop instructions. These additions and removals are the design choices that distinguish a derivative from its base; they are also where the most interesting work in the corpus lives.

**Theming.** Orthogonal to all the above is the surface theme — the cultural skin laid over the instruction set. Cooking (Chef), Shakespearean play (Shakespeare), action-movie quotes (ArnoldC), Schwarzenegger catchphrases (ArnoldC), Chilean slang (La Weá), 19th-century literary Chinese (Wenyan). Theming changes how the language reads without necessarily changing how it executes. It is the most visible design choice and often the one a new author chooses first; it is also, on its own, the cheapest contribution one can make to the corpus.

These eight axes — base machine, memory shape, cell width, instruction count and set, encoding, I/O, additions and removals, theming — between them account for most of the variation across the corpus. They are not the only possible axes, and Babel's schema (next section) will need to be open to additions.

The 2026-05-13 audit against the wider 53-artifact LLM-oriented PL field confirmed this expectation by surfacing further axes the original eight could not cleanly express. The base-machine enum, originally drawn from esolang shapes (tape, stack, OISC, 2D-grid, queue, string-rewriting, combinator), must expand to include procedural-with-named-environment, functional/lambda-calculus, LLM-orchestration, decoder-constraint, and markup/data-format machines if the schema is to describe SimPy, Pel, Quasar, DSPy, SGLang, APPL, LLMON, TOON, XGrammar, and Inflexión. A ninth axis — *artifact kind*, distinguishing an executable PL from a markup language, a data format, a decoder contract, an orchestration DSL, a surface-transformation layer, or an intermediate representation — accounts for whole families of work (LLMON, TOON, XGrammar, Type-Constrained Codegen, B-IR) that fall outside the executable-language assumption baked into the original eight. The original *additions and removals* axis, finally, is doing the work of an entire methodology by itself: it absorbs evaluation strategy, effect and capability model, toolchain targets, observability primitives, verification discipline, ecosystem artifacts, and instruction additions proper. Splitting it into seven sub-axes lets Quasar's reliability bounds, NanoLang's mandatory-tests discipline, DSPy's LLM-call primitives, and SGLang's runtime co-design each become reviewable rather than narrative.

These extensions are formalised in §4; the present section preserves the original eight-axis observation about the esolang corpus that motivated the schema, since the original observation still holds for the field Babel was first designed to describe. The wider field surveyed in the third installment [@rodriguez_llm_oriented_2026] needs more axes; the original eight are the foundation those additions sit on.

## 4. Babel's parameter schema

Babel proposes that a programming language — esoteric or LLM-oriented — can be specified, for the purposes of generation, as a parameter set across the axes inventoried in §3 plus a small number of *meta-parameters* that describe the language's character rather than its mechanism. The schema has expanded over the first four drafts of this paper through the discipline of friction-with-real-languages: friction surfaces gaps; gaps drive extensions; the schema improves through use. The present Draft 4 form is what follows; Draft 4 is the revision that formalises the four extensions (E3, E4, E7, E9) that Draft 3 had deferred pending the empirical-cascade work in [@rodriguez_llm_oriented_2026 §7], the schema audit's E4/E7/E9 data-gathering pass [@empirical_e4_e7_e9_2026], and the four-way verbosity-stratification dataset [@empirical_step1_extended_2026].

The mechanical parameters follow §3, with three additions surfaced by the 2026-05-13 audit:

- *Base machine*, *memory shape*, *cell width*, *instruction count and set*, *encoding*, *I/O model*, *theming* — as in the original eight-axis sketch.
- *Base machine* enum is now expanded beyond esolang-typical shapes to admit **procedural-with-named-environment**, **functional/lambda-calculus**, **LLM-orchestration**, **decoder-constraint**, and **markup/data-format** machines. Memory shape and cell width are *conditional on* base-machine type in the new families: for named-environment machines they collapse into the type-system description; for LLM-orchestration machines they are not applicable.
- A ninth mechanical axis, **artifact kind**, distinguishes the kind of artifact the parameter sheet describes: executable PL, markup language, data format, decoder contract, orchestration DSL, surface-transformation layer, or intermediate representation. The original eight axes implicitly assumed *executable PL*; LLMON, TOON, XGrammar, Type-Constrained Code Generation, and B-IR all sit at other values of this axis.
- The original *additions and removals* axis decomposes into **seven sub-axes** to make the LLM-oriented field's design choices reviewable rather than narrative:
  - *instruction additions proper* (a clipboard register, a randomness primitive, a debug-print)
  - *effect model / capability scope* (which side-effects the language admits and how they are scoped — Quasar's isolated external calls, Wang's algebraic effect handlers, APPL's tracing)
  - *evaluation strategy* (eager, lazy, mixed, decode-time)
  - *toolchain targets* (compile-to-JS-Python-Ruby for Wenyan; compile-to-C for NanoLang; transpile-to-Python for Quasar)
  - *observability primitives* (tracing, profiling, conformal-prediction bounds)
  - *verification discipline* (no-tests, optional-tests, mandatory-tests, proof, typed)
  - *ecosystem artifacts* (LSP, debugger adapter, package manager, REPL)

The meta-parameters are different in kind. They are knobs on what the resulting language *feels like* to read and write, not knobs on its mechanism. The proposed meta-parameter set:

**Complexity** measures the cognitive cost of writing a non-trivial program. A Subleq machine has high complexity (one instruction, large programs); Brainfuck has medium complexity; a richer fungeoid like Befunge has lower complexity per useful program.

**Abstraction** measures the gap between surface syntax and execution model. Brainfuck is low-abstraction (the instructions correspond directly to operations on the tape); Shakespeare is high-abstraction (a soliloquy assigns a value through letter counts and metaphor); Wenyan is high-abstraction in its surface syntax but lower in its underlying semantics.

**Verbosity** measures bytes per logical operation. Chicken (one token, repetition-encoded) is hyper-verbose; Brainfuck is moderate; J or K (single-byte tokens) are hyper-compact.

**Playfulness** measures the language's self-conscious humour, parody, or theatricality. INTERCAL, Chef, ArnoldC, Fetlang are high-playfulness; Subleq is low.

**Unpredictability** measures the degree to which the same program can produce different results across executions, or the degree to which the language's behaviour deviates from the programmer's reading of the source. Whenever (which executes statements when it feels like it) is high-unpredictability; Malbolge is high in a different way (the instruction effect depends on the address); a deterministic Brainfuck is low.

**Naturalness** measures how closely the language's surface syntax follows a natural-language grammar. The single-value framing first proposed here was surfaced as too coarse by the Wenyan decomposition (§6) and by the companion paper's Inflexión decomposition [@rodriguez_inflexion_2026]; a 2026-05-13 audit against the 53-artifact LLM-oriented PL field then surfaced a fourth distinct position. The parameter now resolves into four sub-values reflecting how — and how deliberately — the natural language is engaged or anti-engaged:

- *Vocabulary substitution.* The language's keywords are natural-language words but the grammar is otherwise English-keyworded-PL standard. Mierda, La Weá, Chespirito, BIRL, ArnoldC, and Monicelli sit here.
- *Lexical-grammar engagement.* The language uses natural-language morphology or word order in a partial way: a few inflection-driven syntactic rules over otherwise-standard PL grammar. LOLCODE, Shakespeare, and Rockstar approximately sit here; the position is sparsely populated relative to the other three.
- *Deep-grammar engagement.* The language's parser handles the substrate language's actual grammatical constructions: agreement, case, mood, aspect, clitic systems, particles in their literary positions. Wenyan, Perligata, Tampio, and Inflexión [@rodriguez_inflexion_2026] sit here.
- *Anti-readable-by-design.* The language deliberately abandons the human-readability axis, often in order to optimise LLM-oriented properties unconstrained by human-cognitive-load concerns. B-IR (byte-encoded intent representation) and ALaS (JSON-canonical, LLM-emit-LLVM-IR-only) sit here. The position is increasingly populated in the LLM-oriented PL field [@rodriguez_llm_oriented_2026].

A language can sit at any of the four levels independently of its other parameters. The decomposition fixes the case noted in §6 below: under the old single value, *Inflexión* and *Mierda* were both reported as "naturalness involving Spanish," which obscured a structural difference (Mierda is vocabulary substitution; Inflexión engages grammar). The four-value split makes that distinction parameter-explicit, and adds the human-readability-negative corner that the original three-value sketch missed.

**LLM-friendliness** measures how well the language's surface form serves consumption and generation by large language models. The parameter was not present in the first draft of this paper; it was surfaced by the 53-artifact 2024–2026 LLM-oriented PL design field surveyed in the third installment [@rodriguez_llm_oriented_2026], in which existing artifacts (SimPy, Pel, Quasar, LLMON, and others) make design choices the original six meta-parameters cannot cleanly express. An initial cluster of five sub-values proposed in [@rodriguez_llm_oriented_2026 §6] was refined to three after the 2026-05-13 schema audit observed that two of the five were *outcome metrics* (things a language is measured at) rather than *design parameters* (things a designer chooses). The schema includes only design parameters; outcome metrics move to the empirical methodology described in [@rodriguez_llm_oriented_2026 §7]. The LLM-friendliness parameter therefore resolves into three design sub-values:

- *BPE-token alignment.* Surface tokens are chosen from a target LLM family's tokenizer vocabulary, so that the language's keywords resolve to single BPE tokens. SimPy preserves Python's keyword choices; Pel and Quasar implicitly benefit from this axis; a deliberate LLM-targeted design would optimise it explicitly.
- *Token cost per semantic unit.* Tokens consumed per unit of semantics (one binding, one function definition, one effectful statement). Sun, Du, Yang, Li, and Lo [@sun_simpy_2024] report SimPy at -13.5% versus Python on CodeLlama and -10.4% on GPT-4; a preliminary five-program / six-tokenizer measurement [@rodriguez_llm_oriented_2026 §7] places Inflexión at -28% versus Python on average.
- *Ambiguity load.* Fraction of the language's surface forms whose meaning depends on context the LLM must resolve from prior tokens or from external state. Natural-language-derived syntaxes have higher ambiguity load; typed formal languages drive it toward zero. Type-Constrained Code Generation [@mundler_type_constrained_2025] treats this as a decode-time constraint; LLM-friendly languages can also reduce it at design time. This sub-value pairs structurally with the Draft 4 *enforcement-locus* axis (E4 below), since the choice of where to enforce a contract resolves what the LLM sees as ambiguous; the pairing is captured by treating *ambiguity locus* as a derived subset of the E4 set, rather than as a standalone axis.

LLM-friendliness *outcomes* — generation accuracy and comprehension accuracy — are not parameters; they are what the cascade in [@rodriguez_llm_oriented_2026 §7] measures *for* a given parameter setting. The distinction matters: a designer chooses BPE-token alignment, token cost, and ambiguity load; a benchmark reports the generation and comprehension accuracy that result. Conflating the two would put outcomes inside the design vocabulary and lose the falsifiability the methodology depends on.

The three sub-values are not independent. *Token cost* is partially a function of *BPE-token alignment*; *ambiguity load* interacts with the *encoding* and *theming* mechanical parameters and (via the deep-grammar level of *naturalness*) with the substrate-language's morphological structure. The cluster captures LLM-oriented design intent at a level a new author thinks at, then resolves downward — the same move the original six meta-parameters make for human-oriented design intent.

**Target audience** names whom the language is designed to be read, written, and reasoned about by. The original schema assumed *human-default* throughout; the 53-artifact LLM-oriented PL field makes the assumption no longer safe. The axis admits at least five values, surfaced by audit-decomposition: *human* (Python, Inflexión, Wenyan — designed first for human readers); *LLM-only-by-design* (B-IR, ALaS — explicitly anti-human-readable, optimised for LLM consumption alone); *both* (SimPy — preserves Python's human-readability claim while reducing LLM token cost); *theme-aware humans* (LOLCODE, Shakespeare, ArnoldC — designed for human readers who recognise the cultural register); and *machine-only* (Whitespace, Malbolge — not designed for any human reader to fluently use, but not designed for LLMs either). The audience parameter feeds the LLM-friendliness cluster's design choices: a language with target-audience = LLM-only-by-design optimises BPE-token alignment and ambiguity load aggressively, accepting human-readability losses the human and theme-aware-humans positions would not.

**Grammar regularity** measures how uniform or irregular the language's surface grammar is. The audit surfaced this as a separable axis: Pel and Tokenese deliberately maximise regularity (uniform syntax); Wenyan and Inflexión deliberately engage natural-language irregularity; Python and SimPy sit at a middling regularity inherited from Python's PEP-8-shaped grammar. The values: *uniform* (every form follows one shape), *regular* (small number of exception-free rules), *mixed* (regular core plus irregular surface — most natural-language-derived PLs), *irregular* (rich exceptions inherited from the substrate), *irregularity-engaging* (the irregularities themselves carry semantic load — Inflexión's *ser*/*estar* split is an example). The parameter is not the same as *naturalness*: a deep-grammar-engagement language can have uniform internal grammar (Lojban-style constructed languages) or irregular grammar (any natural language); a vocabulary-substitution language inherits whatever regularity its base has.

The meta-parameters are not independent of the mechanical parameters either. *Verbosity* is partially a function of encoding and instruction count; *naturalness* is partially a function of theming; *LLM-friendliness* is partially a function of encoding and of vocabulary choice; *target audience* feeds and is fed by every parameter. The point of including them anyway is that they capture the design intent at a level a new author thinks at. A new author rarely starts with "I want a one-dimensional unbounded tape with byte-sized cells." They start with "I want something playful, high-verbosity, themed around X, cheap for the target LLM to generate, and primarily readable by humans." The meta-parameters let Babel meet the author at that level and resolve down to the mechanical parameters as a second step.

This two-layer schema is the methodological contribution. The field has had the mechanical axes available implicitly for thirty years; what it has not had is a vocabulary at the level of design intent that can be turned into a working language without each author rebuilding the same scaffold.

**Orthogonality violations and conditional typing.** The two-layer schema survives the Draft 4 extensions but admits structural revisions the audit identified: some mechanical parameters are no longer orthogonal to *base machine*. *Memory shape* and *cell width* are tape-machine concepts that collapse for named-environment, functional, and LLM-orchestration machines; the schema treats them as *conditional on* the chosen base machine rather than independent. *Theming* and *naturalness* lose independence at the deep-grammar end (when the cultural substrate carries semantic load, the two are no longer separable). *Encoding* and the new *tokens-per-op verbosity stratum* (E3 below) are not independent once tokenizer alignment becomes its own axis. *Unpredictability* loses independence from the new *effect model* sub-axis. The schema retains both parameters in each case for descriptive clarity, but a reader filling in a parameter sheet should treat the dependent ones as derivative — set base machine first, set theming and naturalness jointly, set encoding and tokens-per-op together.

**Verbosity stratification (E3).** The single *verbosity* parameter inherited from the original Brainfuck-derivative schema treated bytes per logical operation as a one-dimensional cost. The 2024–2026 LLM-oriented field has stratified the relevant cost axes into four separable measures, each with its own units, its own consumer, and its own design implications, and Draft 4 formalises the stratification accordingly. *Verbosity* now resolves into four sub-values:

- *Bytes-per-op*, the storage / transmission / version-control-diff cost. Identical to the original single-value parameter for ASCII-only languages.
- *Characters-per-op*, the human-read cost. Diverges from bytes when the language uses multi-byte UTF-8 (Wenyan, Inflexión with accented characters).
- *Morphemes-per-op*, a proxy for semantic density of the surface form. Morphologically rich languages (Spanish, Finnish, Classical Chinese) pack more semantic distinctions per morpheme than analytic ones.
- *Tokens-per-op*, the cost the LLM pays per semantic operation, against a chosen tokenizer slate (`tiktoken cl100k_base`, `o200k_base`, the Llama / Qwen / Mistral / GPT-2 tokenizers, and others as licensing permits).

The four strata can decorrelate. The empirical step-1 extended dataset [@empirical_step1_extended_2026] measures the four strata across five short programs in three surface forms (Python, SimPy-style hand-strip, Inflexión) against six tokenizers, and finds that SimPy moves the four strata together (a single number suffices — bytes, chars, morphemes, tokens all drop by between 36% and 52%) while Inflexión decorrelates them (bytes per op essentially tied with Python; morphemes per op +359.9%; tokens per op +15% to +20% on every modern tokenizer; per-program tokens nonetheless −28% on average through fewer ops per program). The companion paper [@rodriguez_llm_oriented_2026 §8] develops the methodological argument that a single verbosity parameter implicitly assumes the four measures correlate; when they do (analytic-language interventions), the parameter is adequate; when they do not (any substrate that engages morphology), the single parameter forces the designer to pick which measure to optimise without realising the choice has been made. Draft 4 lifts the choice to the surface: a designer choosing low verbosity must now say *low on which stratum*.

The four sub-values are not orthogonal. Bytes-per-op and characters-per-op nearly co-vary for ASCII-only languages and differ by a small constant for UTF-8 substrates. Tokens-per-op is conditional on tokenizer choice and shifts with model family. Morphemes-per-op is cross-language-comparable only as an ordinal ranking (the spaCy feature-count proxy used in [@empirical_step1_extended_2026] is more expressive than the English word-count proxy and the ratio is not on a common absolute scale). Draft 4 treats the four sub-values as descriptively independent — fillable independently on a parameter sheet — while documenting these dependencies as conditional typing the schema acknowledges.

**Enforcement locus (E4).** A new mechanical axis names where in an artifact's lifecycle the language contract is enforced — where the system rejects a string as *not in the language*. The audit's data-gathering pass [@empirical_e4_e7_e9_2026] cited twenty-five artifacts across six loci and surfaced one structural refinement worth recording in the schema itself. The six values:

- *Author-time.* Contract enforced as the human (or LLM acting as author) types: linters, formatters, IDE warnings, in-editor type checkers. Diagnostics emit before any parse / compile cycle. Examples: CNL-P's static linter for NL prompts [@xing_cnlp_2025]; NanoLang's LSP-and-DAP-on-launch ecosystem; Bembenek's LMPL'25 position on PL-style design discipline for ad-hoc reasoning tools.
- *Parse-time.* Contract enforced when the source is read into an AST: grammar errors, syntactic rejection, well-formedness. Examples: Python's `ast.parse`; SimPy [@sun_simpy_2024] (formatting-strip-then-Python-parse); TOON's spec parser [@toon_spec_2025]; ALaS's JSON-canonical AST; Wenyan's literary-particle grammar.
- *Compile-time.* Contract enforced by a static analyser between parse and runtime: type checking, borrow checking, effect inference, exhaustiveness, totality. Examples: mypy / Python typing as a gradual blend of parse and compile; Rust's borrow checker; Bayazit and Li's LMPL'25 proof-assistant case study [@bayazit_proof_2025]; NanoLang's mandatory-tests-as-syntax; Pel's uniform-grammar type pass [@mohammadi_pel_2025]; Quasar's lambda-core type / effect check [@mcaleese_quasar_2025].
- *Runtime.* Contract enforced during execution: dynamic type errors, assertion failures, capability violations, effect-handler rejections. Examples: LMQL `where` clauses [@lmql_2024]; APPL's async LLM-call + tracing runtime [@dong_appl_2025]; SGLang's structured-generation runtime [@sglang_2025]; Wang's algebraic effect handlers [@wang_lmpl_2025]; Pangolin's selection-monad effects [@tan_pangolin_2025]; Quasar's capability sandbox for external calls.
- *Decode-time.* Contract enforced by the LLM sampler itself: at each sampling step, tokens that would violate the grammar / type / format are masked from the logit distribution. The contract lives *inside* the decoder, not the language toolchain. Examples: XGrammar [@dong_xgrammar_2024] (CFG masking, vocabulary split, up to 100× speedup); Type-Constrained Code Generation [@mundler_type_constrained_2025] (prefix automata + inhabitable-type search, >50% HumanEval/MBPP compile-error reduction); Park et al.'s CSG quotienting; Park and Kang's Grammar-Aligned Decoding [@park_grammar_aligned_2024] (ASAp algorithm restoring the LLM distribution under CFG masking).
- *Train-time.* Contract internalised into the model weights: fine-tuning, type-guided synthesis, representation redesign. Examples: type-guided synthesis (Mündler-Vechev follow-on); SA-BPE source-attributed tokenizer training; LoRA-adapter-trained encoding of SKI combinators (symbolic compression); the implicit train-time alignment Pel, SimPy, and NanoLang's surface forms bet on through pre-training-distribution match.

The structural refinement: **E4 is set-valued, not single-valued.** Several artifacts enforce contracts at multiple loci simultaneously. LMQL enforces both at decode-time (`where` clauses) and at runtime; NanoLang enforces at author-time (LSP), parse-time, and compile-time (mandatory tests); Quasar enforces at compile-time (type / effect check), at runtime (capability sandbox), and at decode-time (via type-constrained-decoding integration); Type-Constrained Codegen enforces at both decode-time and train-time when paired with type-guided fine-tuning. A parameter sheet may therefore declare *a set of loci* rather than a single locus, and the audit data shows the redundant-enforcement combinations are themselves a design choice worth recording. Draft 4 treats E4 as a set parameter with the six-value enum as the underlying vocabulary.

E4 pairs structurally with the LLM-friendliness *ambiguity load* sub-value: the choice of where to enforce a contract resolves what the LLM sees as ambiguous at any given step of generation. The pairing is captured by treating *ambiguity locus* as a derived parameter — the subset of the E4 set at which ambiguity is reduced — rather than as a standalone axis.

**Unpredictability split (E7).** The original *unpredictability* meta-parameter measured the degree to which the same program could produce different results across executions, with Whenever and Malbolge as the canonical examples. The 2024–2026 LLM-oriented field introduces structurally distinct kinds of unpredictability that share the original parameter's surface but not its mechanism. Draft 4 splits *unpredictability* into four values, three positive-valued and one zero:

- *Programmer-facing chaos.* The language's surface semantics is intentionally unpredictable from the programmer's reading. Even with deterministic hardware and a deterministic interpreter, the language *as designed* makes program behaviour hard to predict from the source. Examples: Whenever; Malbolge; INTERCAL's `PLEASE` politeness threshold and `COMEFROM`; Befunge's self-modifying 2D grid; Brainfuck-derivative variants that randomise step order. This is the original Babel parameter as Draft 3 framed it.
- *Model-call nondeterminism.* Non-determinism enters via LLM calls embedded inside the language's semantics. The host language and surface syntax are deterministic; the variance comes from the model's stochastic sampling at LLM-call sites. Examples: DSPy [@dspy_2025]; SGLang [@sglang_2025]; APPL [@dong_appl_2025]; PDL [@vaziri_pdl_2025]; Plang [@hu_plang_2025]; Pangolin's selection monad over best-of-n / retries [@tan_pangolin_2025]; Wang's algebraic effect handlers [@wang_lmpl_2025]. The audit's recommended call [@empirical_e4_e7_e9_2026]: this category owns any artifact that invokes a model at runtime, regardless of the sampler's nominal determinism.
- *Reliability-bounded nondeterminism.* Non-determinism that comes with statistical bounds: the artifact admits stochastic behaviour but pairs each stochastic result with a calibrated reliability interval (conformal-prediction width, type-correctness probability, generation-pass-rate confidence). The programmer can reason about *how unreliable* a result is, not just *that* it is unreliable. Examples: Quasar [@mcaleese_quasar_2025] (conformal-prediction uncertainty as a first-class language feature); Type-Constrained Code Generation [@mundler_type_constrained_2025] (functional correctness reported with measured rates while type correctness is deterministic at decode-time); type-guided synthesis (train-time guarantee with measured residual error). Sparsely populated; recognisable because the artifact ships a numeric bound. Quasar is the prototype.
- *n/a — deterministic by design.* The artifact has no unpredictability sub-value at all. The audit pass [@empirical_e4_e7_e9_2026] surfaced this value as missing from the original three-way split: most of the 53-artifact LLM-oriented corpus is deterministic by design, and the schema needs the zero. Examples: SimPy [@sun_simpy_2024]; Pel [@mohammadi_pel_2025]; LLMON [@hind_llmon_2026]; TOON; PDL (as a declarative-prompt-language *spec*, not the LLM calls it directly emits); Inflexión [@rodriguez_inflexion_2026]; the entire Brainfuck-derivative family with the noted exceptions above.

A fourth candidate — *environment-driven nondeterminism* (wall-clock dependence, scheduler interleaving, I/O race timing, network latency, RNG seeds, GPU non-associativity, KV-cache state shared across requests) — appears in the field but the audit's recommendation is **not** to add it as a top-level E7 sub-value [@empirical_e4_e7_e9_2026]. The reason is twofold: every executable PL in the corpus inherits some environment-driven nondeterminism from its base machine, which would force a *yes* on almost every executable artifact and an *n/a* on every non-executable artifact (the wrong discrimination); and the cases where environment-driven nondeterminism is *designed in* rather than merely inherited collapse into one of the existing values (designed-in scheduler quirk = programmer-facing chaos; designed-in race over LLM-call results = model-call nondeterminism). Draft 4 documents environment-driven nondeterminism as a noted-but-not-axial inheritance from the base machine, captured by the E1-expanded `base_machine` enum rather than by E7.

**Derivation relation (E9).** A new axis distinguishing stand-alone languages from various host-language-derived relationships. The audit's data-gathering pass [@empirical_e4_e7_e9_2026] categorised the 53 lit-pass artifacts and surfaced two structural refinements. Draft 4 formalises seven values plus an n/a:

- *Stand-alone.* Designed from scratch with no host language. Examples: Pel [@mohammadi_pel_2025]; Quasar [@mcaleese_quasar_2025]; NanoLang; B-IR [@b_ir_2025]; Inflexión [@rodriguez_inflexion_2026]; CNL-P [@xing_cnlp_2025]; MLDS [@mlds_2025]; TOON [@toon_spec_2025]; LLMON [@hind_llmon_2026]; Tokenese [@youvan_tokenese_2025]; the Babel-original esolangs (Whenever, Malbolge, INTERCAL, Mierda, La Weá, Wenyan).
- *Surface-layer-over-X.* The artifact preserves a host language's AST but transforms the surface form. Foundational example: SimPy [@sun_simpy_2024] — same AST as Python, formatting and whitespace stripped. Other examples: Token Sugar [@sun_token_sugar_2025] (cross-PL surface shorthand); Symbolic Compression (SKI-combinator + GAEL encoding via LoRA adapter); ShortenDoc [@shortendoc_2025]; the Brainfuck-derivatives family (theming as surface-layer over the same eight-instruction tape).
- *AST-equivalent-to-X.* The AST itself is the surface. Examples: ALaS (JSON-canonical AST as surface, LLVM-IR backend); B-IR at the borderline (the byte encoding is intentionally AST-equivalent to a notional textual IR).
- *Transpiles-to-X.* The artifact compiles to a target language as a published feature, not just an internal step. Examples: Quasar transpiles to Python; NanoLang compiles to C; Wenyan transpiles to JavaScript, Python, and Ruby; Compiled-AI transpiles from natural language to code at compile-time only; MLDS transpiles meta-language definitions to AI-targetable DSL specs.
- *Embedded-in-host-X.* The artifact is expressed as a sublanguage inside a host language's surface, typically Python. Examples: DSPy (Python decorators and classes) [@dspy_2025]; APPL (Python prompt-string sublanguage) [@dong_appl_2025]; SGLang (Python with co-designed runtime) [@sglang_2025]; PDL (YAML-inside-Python-runtime) [@vaziri_pdl_2025]; LMQL (Python with `where`-clause constraints) [@lmql_2024]; Wang's effect handlers; Pangolin's selection-monad embedding; Plang (NL host with control-flow keyword guest) [@hu_plang_2025].
- *Host-guest-stack.* The host language calls a guest sublanguage with two-language semantics across the boundary. Examples: APPL (Python host + prompt-string guest with async runtime + tracing); SGLang (Python frontend + co-designed inference runtime that is inseparable from the language); DSPy (Python host orchestrates LLM-call guest via compiled prompts); PDL (YAML host + LLM-call guest); LMQL (Python-shaped host + decode-time constraint language as guest). All overlap with *embedded-in-host-X*; the host-guest distinction is whether the two languages are semantically inseparable.
- *Decoder-contract.* **The seventh value the audit surfaced** [@empirical_e4_e7_e9_2026]: the artifact's relation is to the *decoder* of an LLM, not to a host PL. The artifact is not embedded in a host language, does not transpile to a target, is not stand-alone in any conventional sense, and is not a surface layer over an existing AST. It is a PL-shaped contract attached to the sampler. Examples: XGrammar [@dong_xgrammar_2024]; Type-Constrained Code Generation [@mundler_type_constrained_2025]; Park et al.'s CSG quotienting; Park and Kang's Grammar-Aligned Decoding [@park_grammar_aligned_2024]; Geng et al.'s Flexible Grammar-Constrained Decoding. The relation is the structural pair to the E4 *decode-time* enforcement locus: where a decode-time-enforced artifact lives is *in* the decoder, not adjacent to a host PL.
- *n/a — not an artifact with a derivation relation.* The second structural refinement: roughly 30% of the 53-artifact corpus is *about* the field rather than *in* it — surveys, position papers, counter-evidence studies, venues, tokenizer studies (Du-Wang-Wang's PL-techniques position [@du_wang_wang_2025]; Cao, Xie, and Yan's NL-aligned-prover call [@cao_nl_aligned_2025]; the *Let Me Speak Freely* counter-evidence study [@tam_let_me_speak_2024]; the Bayazit-Li proof-assistant case study [@bayazit_proof_2025]; the *Vibe Coding* and LRPL/DSL surveys; LMPL 2025 / LMPL 2026 venue pages [@lmpl_2026_cfp]; the Scaling Laws for Code work [@scaling_laws_code_2026]; SA-BPE tokenizer training [@morphbpe_2025]; the JSON/YAML/Markdown/TOON practitioner benchmarks [@hannecke_format_2025]; the kirancodes "return to mediocrity" essay [@codes_mediocrity_2025]; Xu et al.'s LLM-preferences empirical study [@xu_llm_preferences_2025]; the Kore machine-native essay [@kore_machine_native_2025]). The original schema's framing that *derivation relation* applies to every artifact in the field was a small overshoot; the *n/a* value admits the position-and-survey class formally.

Draft 4 is the present revision. The four extensions formalised here, taken together with the six Draft 2–3 extensions (E1 base-machine expansion, E2 naturalness sub-values, E5 artifact-kind axis, E6 target-audience axis, E8 additions-and-removals decomposition into seven sub-axes, E10 grammar-regularity axis), exhaust the ten extensions surfaced by the 2026-05-13 audit. Further extensions surfaced when the empirical cascade [@rodriguez_llm_oriented_2026 §7] runs Stages 2–4 will be folded into a subsequent revision.

## 5. Three output targets, three readers

A parameter set is not a language. To turn a Babel parameter sheet into a usable artifact, the runtime produces three outputs.

The first is **a runnable interpreter** for the generated language: a small, self-contained program (target language to be decided; Python and Rust are the natural candidates) that reads source files written in the new language and executes them according to the chosen semantics. The interpreter's role is to make the generated language *real*: a language without a working executor is a specification, not a programming language. The interpreter is also the artifact most useful to students and casual users, who can pick it up and run programs without engaging Babel itself.

The second is **a transpiler** that converts source in the generated language to a chosen target language — typically the language of its base machine (Brainfuck for Brainfuck-derivative generations, C or Python for register-machine generations) — so that programs written in the new language can be executed via existing toolchains. The transpiler matters for two reasons. It makes generated languages composable with tools already in the ecosystem; and it provides a second execution path that can serve as a sanity check on the interpreter.

The third is **a specification page** in the format of an esolangs.org wiki entry, giving the language's syntax, semantics, instruction reference, example programs, and design rationale. The specification is the artifact that joins the discourse: it is what other esolang authors read, what wiki maintainers can index, what reviewers cite, and what later installments of this project can refer back to. A generated language without a specification is a private toy.

Three outputs serve three readers — the user who runs programs, the developer who integrates with existing infrastructure, and the writer who joins the wiki community. None of the three is more important than the others; what matters is that the methodology produces all three from one parameter set, in lockstep, without each author having to reproduce the spec by hand from a working interpreter.

## 6. Retrospective decompositions

The methodology can be tested before the runtime exists by decomposing existing esoteric languages into their would-be Babel parameters. If the decomposition is clean — if the parameter sheet for an existing language picks out exactly that language and not its neighbours — the schema is doing useful work. If the decomposition is muddy, the schema needs revision. Five worked decompositions, drawn first from the Spanish-flavoured corner of the corpus and then from the emerging LLM-oriented field.

**Mierda** [@wiki_mierda] is a Brainfuck derivative with Spanish keywords. Its Babel parameter sheet would read: base machine = Brainfuck tape; memory shape = one-dimensional unbounded; cell width = byte; instruction count = 8; instruction set = {`Mas`, `Menos`, `Derecha`, `Izquierda`, `Decir`, `Leer`, `Iniciar Bucle`, `Terminar Bucle`}; encoding = natural-language keyword tokens separated by whitespace; I/O = character; additions = none; removals = none; theme = Spanish vocabulary. Meta-parameters: complexity medium, abstraction low, verbosity medium-high (the natural-language keywords inflate token length), playfulness moderate, unpredictability zero, naturalness *vocabulary substitution*, LLM-friendliness mostly inherited from BF — moderate token cost (the inflated keywords cost LLM tokens), BPE-alignment low (Spanish keywords fragment under English-trained tokenizers), ambiguity load low (the language is keyword-substitution-deterministic). The decomposition is clean: every cell of the parameter sheet is filled in unambiguously, and the resulting parameter sheet picks out Mierda specifically.

**La Weá** [@wiki_lawea] is a Chilean-Spanish Brainfuck derivative with one significant addition — a clipboard register that stores a single byte separately from the tape. Its parameter sheet matches Mierda's on the mechanical axes except: instruction count = 16; instruction set = sixteen Chilean-slang tokens; additions = clipboard register with two associated instructions (store-to-clipboard, recall-from-clipboard); theme = Chilean dialect. Meta-parameters: similar to Mierda but with higher playfulness (the dialect register carries comic weight) and *vocabulary substitution* naturalness at the dialect end of the scale (Chilean slang is further from "neutral" Spanish vocabulary, which makes it *more* culturally specific even though no grammar is engaged). The decomposition picks La Weá out from Mierda specifically by virtue of the *additions* axis (clipboard register) and the dialect-specific *theme*.

**Wenyan** [@huang_wenyan] is harder and was the case that first surfaced the *naturalness* coarseness now formalised in §4. Its base machine is a procedural / variable-based machine with named variables, conditionals, loops, and functions — closer to a stripped-down ALGOL family than to a tape machine. Memory shape = named-variable model with no fixed shape; cell width = arbitrary-precision integer or string; instruction count = several dozen, organised into syntactic categories (declarations, control flow, arithmetic) rather than primitive operations; encoding = Classical Chinese characters and particles (者, 也, 而) in their literary positions; I/O = whole-line via standard input/output; additions = compilation to JavaScript, Python, and Ruby targets; theme = Classical Chinese literary register. Meta-parameters: complexity moderate, abstraction high, verbosity moderate, playfulness moderate (the language is taken seriously by its community), unpredictability low, naturalness **deep-grammar engagement** (the parser handles real Classical Chinese grammar, not vocabulary substitution). LLM-friendliness: BPE-alignment is variable per tokenizer (CJK tokenization differs by model family); token cost moderate; generation accuracy is untested but Classical-Chinese training-data exposure is thin in most frontier models, so the floor is low.

**Inflexión** [@rodriguez_inflexion_2026] is the case that confirms the deep-grammar level needs to be a level rather than a binary. Its parameter sheet under the old single-value naturalness would have collided with Wenyan; under the three sub-values it picks out the same level but a different substrate language. Base machine = procedural / variable-based with explicit mutability typed through the *ser* / *estar* split; memory shape = named-variable; cell width = arbitrary-precision / typed; instruction count and set = the six grammatical-semantic mappings of §3 in the companion paper, each carrying one Spanish grammatical feature with a programming-semantic role; encoding = Rioplatense Argentine Spanish surface forms with `.infl` file extension; I/O = character (via *decí*, vos imperative); additions = the six mappings as joint feature set; theme = Rioplatense cultural register. Meta-parameters: complexity moderate, abstraction high, verbosity low (morphological packing carries semantic distinctions without expansion), playfulness moderate, unpredictability zero, naturalness **deep-grammar engagement**, LLM-friendliness: BPE-alignment mixed (frequent Spanish forms tokenize cheaply, rare inflections fragment), token cost surprisingly low (preliminary measurement: -28% versus Python on a small suite), generation accuracy uncharacterised, ambiguity load high relative to typed PLs but mediated by morphology. The decomposition picks Inflexión out from Wenyan specifically via *substrate language* (Rioplatense Spanish versus Classical Chinese), *additions* (the *ser* / *estar* / mood / aspect / clitic / diminutive feature set), and *encoding* (vos imperatives, enclitic pronouns).

**SimPy** [@sun_simpy_2024] is the first LLM-oriented decomposition the schema needs to express, and the decomposition demonstrates that the new LLM-friendliness cluster and the Draft 4 stratifications are doing real work. Base machine = same as Python (a stack-and-frame procedural machine with full object model); memory shape, cell width, I/O all inherited from Python; instruction set = Python's, unchanged at the AST level; encoding = Python source with formatting and whitespace tokens stripped; additions = bidirectional Python ↔ SimPy transformation. Meta-parameters: complexity high (inherited from Python), abstraction high, verbosity intentionally low (the point), playfulness none (Sun et al. are deadly serious about token economy), unpredictability *n/a — deterministic by design* (the E7 sub-value Draft 4 adds; SimPy preserves Python's determinism), naturalness none (Python's grammar is not natural-language-derived; SimPy doesn't change that). LLM-friendliness, finally, is where the design intent lives: BPE-alignment unchanged from Python; token cost per semantic unit reduced by 10–15% on frontier tokenizers (the published result); ambiguity load unchanged from Python. The E3 verbosity stratification specifically captures what the original single-value parameter could not: SimPy reduces *all four* strata together — bytes per op −52.1%, characters per op −52.1%, morphemes per op −36.2%, tokens per op −38% to −46% across the six-tokenizer slate [@empirical_step1_extended_2026; @rodriguez_llm_oriented_2026 §8]. The four-way correlation is itself a design signal: SimPy is the analytic-language intervention whose strata move together, which is why a single number describes it adequately *if* the reader knows the strata correlate. The new E4 axis pins the enforcement locus precisely: SimPy enforces at *author-time* (PEP-8 formatting strip as a linter pass) and at *parse-time* (the stripped form must still parse as Python); the bidirectional transformer enforces nothing the language does not already enforce. The new E9 axis places SimPy at *surface-layer-over-X* with X = Python: AST preserved, encoding transformed. *Every* design move SimPy makes is now captured in five parameters: verbosity (changed, with the four-way stratification making the correlation visible), token cost (changed), enforcement locus (set-valued: author-time + parse-time), derivation relation (surface-layer-over-Python), and unpredictability (n/a). The outcome metrics SimPy reports — pass-rate parity with Python on HumanEval, −13.5% / −10.4% tokens — are not parameters in the decomposition; they are measurements of what the parameter choices produce, and they live in the empirical cascade described in the companion paper [@rodriguez_llm_oriented_2026 §7], not in this section.

**Quasar** [@mcaleese_quasar_2025] is the sixth decomposition the schema needs to express and the one that demonstrates the Draft 4 stratifications working *jointly*. Base machine = lambda-calculus core with isolated external calls (the E1-expanded *functional/lambda-calculus* base-machine value); memory shape = environment-typed lambda binding (collapsed onto the base machine per the orthogonality-violation note in §4); cell width = arbitrary-precision values; instruction set = lambda-calculus primitives plus a small set of isolated-external-call constructs; encoding = ASCII source with type and effect annotations; I/O = effectful primitives gated by capability sandbox. The additions-and-removals decomposition (E8): effect model = capability sandbox for external calls; evaluation strategy = mixed (with auto-parallelisation); toolchain target = Python transpile path (a published feature, not an internal step); observability primitives = conformal-prediction uncertainty bounds as first-class language constructs; verification discipline = compile-time type / effect check; ecosystem artifacts = transpiler and runtime, no LSP at the time of writing.

The Draft 4 axes pick Quasar out unambiguously from the rest of the LLM-oriented field. **E4 (enforcement locus) is set-valued**: Quasar enforces at *compile-time* (type and effect check on the lambda core), at *runtime* (capability sandbox traps unauthorised external calls), and at *decode-time* in the published integrations with type-constrained-decoding work [@mundler_type_constrained_2025]. No single-locus value captures Quasar's contract surface. **E7 (unpredictability split) places Quasar at *reliability-bounded nondeterminism***: Quasar admits stochastic LLM-call results but pairs each with a calibrated conformal-prediction interval, which is the design move that defines this sub-value. Quasar is the prototype the E7 split was added to admit; without the reliability-bounded value, the original three-way split would have forced Quasar into *model-call nondeterminism* alongside DSPy and SGLang, obscuring the structural difference (Quasar's reliability bounds are part of the language; DSPy's are not). **E9 (derivation relation)** places Quasar at *transpiles-to-X* with X = Python — the published transpile path is the relation, distinct from SimPy's *surface-layer-over-X* (different AST in Quasar's case) and from DSPy's *embedded-in-host-X* (Quasar is not embedded in Python; it transpiles to it). LLM-friendliness, finally: BPE-alignment optimised through deliberate keyword choice; token cost per semantic unit competitive with Python on ViperGPT workloads (the published 42% time win is partly a token win and partly an inference-time-coordination win — the parameter sheet captures only the token half); ambiguity load low (the type / effect system drives it toward zero at compile time; the decode-time integration drives the residual to zero at generation time). Verbosity decomposed into the four E3 strata: bytes-per-op moderate (lambda-calc surface is compact); morphemes-per-op low (English keywords, English-derived identifiers); tokens-per-op competitive with SimPy on the ViperGPT corpus per the published reports, though we are not aware of independent measurement across the six-tokenizer slate this paper's companion uses [@rodriguez_llm_oriented_2026 §8 / @empirical_step1_extended_2026]. The Quasar decomposition demonstrates that *set-valued E4* + *reliability-bounded E7* + *transpile-to-X E9* + *stratified E3 verbosity* together pick out a corner of the design space that none of the single-valued or single-stratum framings could express. The decomposition is the kind of test the Draft 4 schema is meant to pass.

Six decompositions, three of which (Mierda, La Weá, Wenyan) ran cleanly against the original six meta-parameters, two of which (Inflexión, SimPy) required the Draft 3 sub-parameters (naturalness sub-values, LLM-friendliness cluster), and one of which (Quasar) required the Draft 4 stratifications (E3 verbosity strata, E4 set-valued enforcement locus, E7 reliability-bounded unpredictability, E9 transpile-to-X derivation relation). The methodology is doing what a methodology should do: friction with new languages surfaces gaps; gaps drive schema extensions; the schema improves through use rather than through theorising.

A 2026-05-13 audit (`plans/babel-schema-audit-2026-05-13.md`) decomposed eighteen artifacts against the schema — the three Spanish-flavoured and the two LLM-oriented decompositions above plus thirteen drawn from the wider 53-artifact LLM-oriented PL field, with Quasar's decomposition above as the audit's most demanding case. The audit surfaced ten schema extensions and five orthogonality violations. **All ten extensions are now formalised in Draft 4.** The Draft 2 revision admitted E2 (naturalness sub-values) and E10 (LLM-friendliness cluster, originally numbered as part of the meta-parameter set). The Draft 3 revision added E1 (`base_machine` enum expansion), E5 (`artifact_kind` axis), E6 (`target_audience` axis), and E8 (`additions_and_removals` decomposition into seven sub-axes). The present Draft 4 revision adds the remaining four — E3 (verbosity stratification), E4 (enforcement locus, set-valued), E7 (unpredictability split with the *n/a — deterministic by design* value), and E9 (derivation relation with the *decoder-contract* value and the *n/a* class for position / survey / counter-evidence artifacts). The four Draft-4 extensions are formalised in §4 above and tested against the SimPy and Quasar decompositions in this section.

The five orthogonality violations the audit identified — `memory_shape` × `base_machine`, `cell_width` × `base_machine`, `theming` × `naturalness` (at the deep-grammar end), `encoding` × the new E3 tokens-per-op stratum, and `unpredictability` × the E8 effect-model axis — are documented in §4 as conditional-typing constraints rather than as schema breaks. The two-layer mechanical-plus-meta structure of the schema survives them. Draft 4 acknowledges the conditional dependencies explicitly: a parameter sheet sets `base_machine` first; `memory_shape` and `cell_width` are then conditional on the machine-type choice; `theming` and `naturalness` are set jointly when the substrate engages deep grammar; `encoding` and the tokens-per-op stratum are set together when tokenizer alignment is load-bearing; and the *unpredictability* sub-value reduces to *programmer-facing chaos* once the E8 effect-model decomposition absorbs the model-call and environment-driven variants.

Further extensions surfaced when Stages 2–4 of the empirical cascade [@rodriguez_llm_oriented_2026 §7] run will be folded into a subsequent revision; Draft 4 is, by construction, complete with respect to the 2026-05-13 audit.

## 7. A thought-experiment forward example

The reverse direction — starting with a parameter sheet and reading off what Babel would produce — is harder to demonstrate without the runtime existing. A thought experiment will have to do.

Suppose an author wants a small esoteric language with the following design intent. *Themed around the Argentine Rioplatense dialect, with voseo. Brainfuck-derivative base for ease of implementation. Eight instructions, vocabulary-substitution naturalness only (no grammar engagement; that is the companion language's territory). Moderate playfulness. Add one instruction that reflects something distinctive about the dialect: the diminutive suffix as a "halve the current cell" operation.*

The resulting Babel parameter sheet:

```
base_machine:    brainfuck_tape
memory_shape:    1d_unbounded
cell_width:      byte
instruction_count: 9
instruction_set:
  ">":  che          # advance pointer (Argentine vocative)
  "<":  vení         # retreat pointer (vos imperative)
  "+":  más          # increment
  "-":  menos        # decrement
  ".":  decí         # output (vos imperative)
  ",":  escuchá      # input (vos imperative)
  "[":  mientras     # loop start
  "]":  ya           # loop end (Rioplatense particle)
  HALVE: ito         # halve current cell (diminutive suffix)
encoding:        whitespace_separated_tokens
io:              character
additions:       [halve_via_diminutive]
removals:        []
theme:           rioplatense_argentine
meta:
  complexity:     medium
  abstraction:    low
  verbosity:      medium
  playfulness:    moderate
  unpredictability: zero
  naturalness:    vocabulary
```

Babel's three outputs would then be: an interpreter that reads source files containing the nine tokens above and executes them against a byte tape; a transpiler from this language to vanilla Brainfuck (with `ito` lowering to a small loop that halves the cell); and a specification page giving the syntax, the instruction reference, two or three example programs ("Hello, mundo," a Fibonacci routine, an echo loop), and a design rationale paragraph.

The example is deliberately modest. The point is not that this language is interesting on its own — it is a vocabulary-skin Brainfuck derivative with one minor addition, and the existing corpus already contains many such — but that *its parameter sheet is short, fillable, and unambiguous*, and that the three outputs follow from it mechanically. A new author who wanted to produce this language hand-rolled would spend an evening writing a small interpreter; if they wanted a transpiler and a wiki page they would spend a second evening. With Babel they fill in a parameter sheet, and the three outputs are produced together, in lockstep, without ever drifting out of sync.

## 8. Educational and academic value

The methodology has value for three audiences at three levels of engagement.

For *students*, Babel makes the field's variation axes legible without requiring the student to first build a working interpreter from scratch. A class on programming-language design can walk through the parameter schema as a taxonomy of design choices, with examples drawn from the corpus, and assign a project that is no longer "build an esoteric language" — a vague brief that produces uneven work — but "fill in this parameter sheet, run it through Babel, and write a one-page rationale for your choices." The student learns the same material and produces a comparable artefact, but the focus is on the *choice of parameters*, which is where the design thinking actually lives. The work the student saves on plumbing is reinvested in the work that matters.

For *researchers*, Babel provides a shared vocabulary for talking about what makes one esoteric language different from another. The current literature on esolangs — such as it is — relies on prose comparison and intuition. A parameter schema gives reviewers, citation-writers, and survey-paper authors a more compact way to summarise a language's design. It is the same kind of contribution that the *features* of typological linguistics make to comparative grammar work: not a substitute for careful prose, but a scaffold on which careful prose can be built more efficiently.

For *language designers* working in adjacent territory — not pure esolang authors, but people designing constrained languages, controlled natural languages [@attempto_ace; @nelson_inform7], domain-specific languages with deliberate strangeness, or pedagogical mini-languages — Babel's schema provides a sanity check on which axes have already been varied in the esolang corpus and which combinations are unexplored. A designer who finds their parameter sheet matching an existing esolang exactly is reinventing; one whose sheet does not match anything is exploring.

The field's historical literary precedents — Cheswick's narration of a security incident as a story [@cheswick_berferd_1992], Raymond's edition of the Jargon File [@raymond_jargon] as a vocabulary that does cognitive work, Kidder's portrait of a minicomputer team as a story of human effort [@kidder_soul_1981] — all suggest that technical artefacts are at their most useful when they are accompanied by the *language to talk about them*. Babel is, in part, an attempt to give the esolang field that language.

## 9. What this is not

Three things Babel is explicitly not.

It is not a production-language toolkit. The interpreter and transpiler outputs are intended for use, not for performance. There is no JIT, no optimisation, no profiling support, and no debugger beyond what the generated language's own semantics provide. The argument is not that production-grade tooling is undesirable; it is that production-grade tooling is the *next* problem, and conflating it with the methodology problem would muddy both.

It is not a replacement for hand-craft. Some of the most interesting languages in the corpus — Malbolge, Piet, Velato, Folders, Vigil — depend on creative leaps that no parameter schema can capture. Malbolge's instruction-effect-dependent-on-memory-address mechanism is not a value of any reasonable parameter; it is an idea. Babel is for the languages that *are* parameter-driven (which, per the corpus, is most of them), and it is meant to free up author energy for the ideas that aren't.

It is not finished. This is the first installment. The parameter schema as presented has at least one known limitation (the *naturalness* axis is too coarse, per §6) and presumably others that will surface when the runtime is implemented. The methodology will improve through use. A first installment that pretended to be complete would be both dishonest about its stage and unhelpful to whatever future author tries to push the work forward.

## 10. What comes next

The companion paper [05] develops one specific instantiation of the design space Babel proposes to parameterise: a hand-built esoteric language that engages Spanish grammar — number, mood, aspect, *ser* / *estar*, clitic ordering, diminutive morphology — as the source of programming semantics rather than as theme. The companion paper joins a small lineage of inflection-driven non-English natural-language esolangs (Perligata in Latin, the unimplemented Espro in Esperanto, Tampio in Finnish, Wenyan in Classical Chinese) and is the first in that lineage to use a living Romance language [@conway_perligata_2000; @espro_2015; @hauhio_tampio; @huang_wenyan; @wiki_lawea]. That language is not generated by Babel and is not constrained by Babel's parameter schema; the choice to hand-build it was made to keep the language design free of any tooling limits not yet discovered. Once both Babel and the companion language exist as working artefacts, a future installment can ask whether Babel's parameter schema can express the companion language. The answer, whatever it is, will be informative about the schema.

Future installments of this paper will address: the *naturalness* sub-parameters to handle vocabulary, lexical-grammar, and deep-grammar engagement separately; an empirical comparison of hand-rolled and Babel-generated language production effort, sufficient to support or refute the order-of-magnitude claim in §1; expansion of the parameter schema to cover language features the schema currently flattens, such as compilation targets and multi-pass interpreters. A first vertical-slice runtime ships with this installment as a Python implementation: parameter sheets generate working interpreters, demonstrated by BF "Hello World" running through both vanilla and Rioplatense parameter sheets. A small browser playground at `babel.roderickc.com` exposes the runtime to readers without a local install: pick a parameter sheet (vanilla Brainfuck or Rioplatense Brainfuck), type a program, watch it run.

The series is the unit of work. This first installment is meant to be readable on its own, but its main value is in establishing the methodology and the vocabulary the rest of the series will build on.

## 11. A closing note

Esoteric programming languages are usually read as a joke first. Read again, slowly, they are a research programme on what mainstream programming languages quietly assume — about syntax, about state, about how a program is supposed to communicate with the person reading it. The two readings are not in conflict. The joke is real, and the research programme is real.

The field has carried the research programme for thirty years on the energy of individual practitioners working in parallel, mostly without inheriting each other's tools. That work has produced an enormous corpus and proportionally little methodology. Babel is offered as a methodology — specifically, a methodology that lets the field's tacit variation axes be made explicit, parameterised, and turned into working languages with three outputs in lockstep. The contribution is not a new esoteric language. It is *a technique for making esoteric languages*, of the kind a museum acquires only after it has accumulated enough specimens for the curation to start mattering.

A methodological note worth carrying through future installments: this paper's approach to its own novelty claims has been *honest preparation* — writing down where pushback is expected and looking for the precedents that would falsify the claims before review rather than after. The lineage acknowledged in §10 surfaced through exactly this practice: a pre-publication literature check intended as confirmation found four prior authors whose work shifted the companion paper's framing from *first* to *fifth in a small lineage*. The same discipline applies forward: any author using Babel whose parameter sheet matches an existing entry is reinventing rather than extending, and the schema's value depends on authors being willing to discover that before the wiki community does.

A wider observation, treated more fully in the LinkedIn series accompanying this paper: technical vocabulary in mainstream programming languages is a deliberate register that strips cultural and historical context from the words it borrows — *daemon* loses Hesiod's *daimon*, *cookie* loses the bakery, *fork* loses both cutlery and the road. The strip-down is useful but not free: the weight remains *available* to a careful reader and *invisible* to a casual one. The esoteric-language field has often moved in the opposite direction, letting cultural weight be visible at the cost of portability. Babel's *theming* axis (§3) names this design choice explicitly so it is no longer silently inherited; the companion paper [05] develops the move further, building a language whose grammar — not only vocabulary — engages cultural specificity.

The next eight hundred attempts can be different from the last eight hundred.
